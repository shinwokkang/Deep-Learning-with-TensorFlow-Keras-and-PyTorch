이전 강의에서 우리는 MNIST 데이터를 다운로드하고 그 형태를 시각적으로 확인했습니다. 이번 영상에서는 본격적으로 데이터를 **'딥러닝 모델이 소화할 수 있는 완벽한 형태'**로 가공(Pre-processing)하고, 케라스(Keras)를 이용해 얕은 신경망 아키텍처를 조립한 뒤, 모델을 직접 학습시키고 평가하는 전체 파이프라인(End-to-End Pipeline)을 완성합니다.

## 🛠️ 1. 데이터 전처리 1단계: 형태 변환과 타입 변경 (Flatten & Type Cast)

> [!NOTE] 
> **전처리(Preprocessing)란 무엇인가요?**
> 현업 실무에서 수집되는 원본 데이터(Raw Data)는 기계가 바로 이해할 수 없는 형태(다양한 크기, 노이즈, 잘못된 타입 등)를 띠고 있습니다. 기계(모델)가 데이터를 올바르게 소화하고 학습할 수 있도록, 사람이 중간에서 데이터의 **형태, 크기, 타입**을 예쁘게 다듬어주는 모든 사전 작업 과정을 '전처리'라고 부릅니다. 실무에서는 전체 프로젝트 시간의 80%가 전처리에 소모될 만큼 가장 중요합니다.

신경망 모델에 데이터를 집어넣기 전, 우리는 2가지 물리적인 전처리 변환을 거쳐야 합니다.
*(**💡 [현업 실무자의 시선]** 실무에서 이미지 전처리는 단순히 크기를 펴는 것으로 끝나지 않습니다. 자율주행이나 의료 이미지 등에서는 이미지를 자르고(Crop), 회전시키고(Rotate), 밝기를 통일하고, 노이즈를 제거하는 등 데이터 증강(Data Augmentation) 파이프라인(예: `ImageDataGenerator` 또는 `tf.data`)을 구축하여 학습 모델이 실전에서도 당황하지 않게끔 품질을 극대화합니다.)*

### 1.1 평탄화 (Flattening: 28x28 -> 784)
우리가 만들 모델은 '얕은 신경망(Dense Layer만 사용)'입니다. 따라서 2차원 정사각형 형태의 이미지를 통째로 받을 수 없으며, 1차원의 얇고 긴 배열로 펴주어야 합니다.

```python
X_train = X_train.reshape(60000, 784).astype('float32')
X_valid = X_valid.reshape(10000, 784).astype('float32')
```

### 1.2 타입 변경 (Int -> Float32)
*   원본 픽셀 값은 `uint8`(0~255 사이의 정수)입니다. 하지만 딥러닝은 소수점 아래 미세한 가중치(Weight)를 곱하고 나누는 '미분(Calculus)' 수학을 기반으로 합니다. 정수형인 채로 소수점 연산을 하면 값이 깎이거나 에러가 납니다. 따라서 소수점을 정밀하게 표현할 수 있는 `float32`(32비트 실수형)로 그릇을 바꿔줍니다.

## ⚖️ 2. 데이터 전처리 2단계: 정규화와 원-핫 인코딩

### 2.1 스케일링 (Normalization: 0~255 -> 0~1)과 기울기 소실(Gradient Vanishing)

모든 이미지 픽셀 값을 255로 나누어주는 스케일링 작업입니다.
```python
X_train = X_train / 255
X_valid = X_valid / 255
```

이 작업은 이전 1.5강에서 다루었던 **'기울기 소실(Gradient Vanishing) 폭탄'**을 제거하는 핵심 과정입니다. 기울기 소실이란 무엇일까요?
기계가 학습한다는 것은 "가중치(Weight)를 업데이트하는 미분 기울기(Gradient)를 구하는 것"입니다. 하지만 입력값이 너무 크면, 활성화 함수(시그모이드)를 통과할 때 값이 극단으로 치우치면서 **미분 기울기가 완벽한 0(평행선)이 되어 기계가 학습을 영원히 멈추는 현상**을 말합니다.

아래 다이어그램을 통해 전처리를 안 했을 때와 했을 때의 치명적인 차이를 직관적으로 확인해 보겠습니다.

<img width="574" height="609" alt="image" src="https://github.com/user-attachments/assets/361d2812-b971-4cd5-8967-bf61c7ce2f0d" />

*   **실무적 접근**: 실무에서도 숫자 데이터(집값, 나이, 연봉 등)가 수만~수억 단위를 넘나들 경우, 아무리 좋은 딥러닝 모델을 짜도 학습이 진행되지 않습니다. 따라서 **반드시 스케일링(Scaling)** 전처리를 거쳐야 모델이 제 성능을 발휘할 수 있습니다.
    *   **💡 [Tip] 수억 단위의 연봉 데이터도 똑같이 수억(Max 값)으로 나누나요?**
    *   **아닙니다!** 이미지의 경우 픽셀의 최대 밝기가 255로 완벽하게 고정되어 있으므로 그냥 255로 나누는 '최소-최대 스케일링(Min-Max Scaling)'이 가장 완벽한 방법입니다.
    *   하지만 실무의 연봉/집값 데이터는 다릅니다. 평범한 직장인들(3천만~5천만 원) 사이에 **'일론 머스크(연봉 10조)' 같은 극단적인 이상치(Outlier)**가 한 명이라도 껴있으면, 전체 데이터를 10조로 나누게 되어 평범한 사람들의 연봉 데이터는 전부 0.000001 처럼 0으로 찌그러져 버립니다. (데이터의 패턴이 소멸함)
    *   **해결책**: 따라서 실무 정형 데이터(표 데이터)에서는 단순히 최댓값으로 나누지 않고, 집단의 평균을 0으로 맞추고 표준편차를 1로 맞추는 **'표준화(Standardization, Z-score)'** 기법이나, 아예 극단치(상/하위 25%)를 무시해 버리는 **'로버스트 스케일링(Robust Scaler)'** 기법을 주로 사용하는 것이 업계 표준입니다.

### 2.2 원-핫 인코딩 (One-Hot Encoding)
사람에게 정답은 그냥 정수 '7'입니다. 하지만 기계에게 "야, 이 사진은 7이야"라고 가르치면, 기계는 7이라는 숫자의 수학적 크기(7은 1보다 7배 크고 무겁다)로 오해하게 됩니다. 

```python
from tensorflow.keras.utils import to_categorical
n_classes = 10
y_train = to_categorical(y_train, n_classes)
```

이러한 기계의 수학적 오해를 막기 위해, 우리는 정수 '7'을 **"나머지는 전부 0이고, 7번 인덱스만 1인 10칸짜리 스위치 표"**로 변환합니다. 이것이 뜨거운(Hot) 곳이 단 한(One) 곳뿐이라는 뜻의 **원-핫 인코딩**입니다.

## 🏗️ 3. 신경망 조립하기 (Sequential API)

이제 전처리된 데이터를 받아먹을 딥러닝 **'모델 껍데기(아키텍처 설계도)'**를 만듭니다. 여기서 껍데기란, 아직 기계가 데이터를 보고 공부(학습)하기 전 상태의 빈 집 구조를 뜻합니다.

```python
from tensorflow.keras.models import Sequential
from tensorflow.keras.layers import Dense

model = Sequential()
model.add(Dense(64, activation='sigmoid', input_shape=(784,)))
model.add(Dense(10, activation='softmax'))
```

이 짧은 코드가 실제로 어떻게 로직이 흘러가는지 다이어그램으로 명확하게 파악해 보겠습니다.

<img width="512" height="590" alt="image" src="https://github.com/user-attachments/assets/5758188f-3609-4de4-8da3-dc134289d22b" />

*   **Sigmoid (시그모이드)**: 직선적인 선형 연산(곱하고 더하기)을 곡선(비선형)으로 휘어주는 역할을 합니다. 비선형성이 추가되어야 기계가 복잡한 패턴을 학습할 수 있습니다.
*   **Softmax (소프트맥스)**: 기계가 마지막에 뱉어낸 점수가 `[100점, 20점, -50점]`처럼 제멋대로일 때, 이를 비율(확률)로 환산하여 `[85%, 14%, 1%]`처럼 다 합쳐서 100%가 되도록 둥글게 깎아주는 궁극의 확률 변환기입니다.

## 🧠 4. 모델 컴파일과 학습 (Compile & Fit)

모델 껍데기를 만들었으니, 이 빈 껍데기가 **'어떻게 공부할 것인지(채점 기준과 공부법)'**를 정해주고(`compile`), 실제 문제집을 주며 훈련(`fit`)을 시작합니다.

### 4.1 Compile (공부법 세팅)
```python
model.compile(loss='categorical_crossentropy', 
              optimizer=SGD(learning_rate=0.1), 
              metrics=['accuracy'])
```
*   `loss`: 틀렸을 때 때리는 회초리(비용 함수)입니다.
*   `optimizer`: 회초리를 맞고 나서 정답에 가깝게 나아가는 튜너(최적화 알고리즘)입니다.
*   `metrics`: 우리 인간이 보기 편하도록 정확도를 퍼센트(%)로 띄워달라는 옵션입니다.

### 4.2 Fit (실전 훈련 시작)과 Epoch / Batch Size의 차이점
초보자들이 가장 헷갈려 하는 두 개념을 직관적으로 분리해 보겠습니다.
```python
model.fit(X_train, y_train, 
          batch_size=128, epochs=200)
```

<img width="505" height="404" alt="image" src="https://github.com/user-attachments/assets/98906d94-0398-4ccd-9a20-a99e0f1a6716" />


*   **배치 사이즈 (`batch_size=128`)**: 한 번에 몇 문제를 풀고 나서 정답을 맞혀보고(채점) 뇌를 업데이트할 것인가? 한 번에 6만 문제를 다 풀고 채점하려면 뇌(RAM) 용량이 터져버리기 때문에 128문제씩 잘라서 풉니다.
*   **에포크 (`epochs=200`)**: 전체 6만 문제짜리 문제집을 처음부터 끝까지 총 몇 번 반복해서 복습(N회독) 할 것인가?

predictions = model.predict(valid_0)

## ✍️ 6. 핵심 요약 및 실전 이해도 점검 (Beginner to Pro)

**[핵심 요약]**
1. **정규화(Normalization) 필수**: $0 \sim 255$의 이미지 픽셀 값을 그대로 딥러닝에 넣으면 기울기 소실 폭탄을 맞고 망가집니다. 반드시 255로 나누어 $0 \sim 1$ 사이의 값으로 전처리해야 합니다.
2. **원-핫 인코딩(One-Hot Encoding)**: 숫자를 '분류'하는 문제에서 정답지(y)는 단순한 정수가 아니라, 정답 인덱스만 1이고 나머지는 0인 평등한 독립 스위치 배열로 변환되어야 합니다.
3. **학습의 본질 (Fit)**: 모델 껍데기를 만들고, `batch_size`(1회 채점 분량)로 쪼개어 가며 수백 번 복습(`epochs`) 시키는 것이 딥러닝 훈련의 알파이자 오메가입니다.

**🤔 실전 점검 질문 (비즈니스 시나리오):**
당신은 자율주행 자동차 회사의 카메라 인식 팀에서 일하고 있습니다. 앞차와의 거리를 예측하는 모델이 아닌, 전방에 표지판이 **"정지(0), 속도제한(1), 직진(2)"** 중 어떤 것인지 '분류'해야 하는 모델을 만들고 있습니다. 
주니어 개발자인 팀원이 당신에게 코드를 가져왔는데, 전처리 코드에 `to_categorical` (원-핫 인코딩)을 쓰지 않고 표지판 정답 데이터를 그냥 0, 1, 2 정수형 그대로 출력층에 넣어버렸습니다. 

Q1. 원-핫 인코딩을 하지 않은 채로 학습을 돌리면 기계(신경망)는 이 0, 1, 2라는 숫자를 수학적으로 어떻게 끔찍하게 오해하게 됩니까? 
Q2. 이 오해 때문에 "정지(0) 표지판과 직진(2) 표지판의 중간은 속도제한(1) 표지판이다"라는 말도 안 되는 결론을 도출할 수 있습니다. 기계의 이 오해를 바로잡기 위해 원-핫 인코딩이 어떻게 이 문제를 해결해 주는지 논리적으로 설명해 보세요.

---

### 💡 실전 점검 질문 모범 답안 

*   **모범 답안 (Q1)**: 기계는 정수 0, 1, 2를 각각 독립적인 이름표로 인식하지 못하고, **크기와 서열이 존재하는 '연속적인 수치'**로 오해하게 됩니다. 기계 입장에서는 "직진(2) 표지판이 정지(0) 표지판보다 수학적으로 2배 더 크거나 무겁구나!"라고 엉터리 기하학적 잣대를 들이대게 됩니다.
*   **모범 답안 (Q2)**: 원-핫 인코딩을 적용하면 정지, 속도제한, 직진은 각각 `[1,0,0]`, `[0,1,0]`, `[0,0,1]`이라는 벡터로 변환됩니다. 이 세 벡터는 크기와 거리의 차이가 단 1도 없는, **완벽하게 동등하고 독립적인 카테고리 스위치** 형태를 갖추게 되어 기계가 카테고리의 본질을 완벽하게 이해할 수 있습니다.

## 🔥 [전공자/전문가용] 심화 보충 설명 (Deep Dive: 미니 배치와 실무 아키텍처)

초보자 코스를 마쳤으니, 케라스의 추상화된 메서드(`fit`) 이면에서 실제로 어떤 잔혹한 컴퓨팅 리소스 전투가 벌어지고 있는지 딥다이브 해보겠습니다.

### 1. 극단적인 배치 사이즈(Batch Size)의 치명적인 한계점
만약 미니 배치(`batch_size=128`)를 쓰지 않고 극단적인 값을 쓰면 어떤 문제가 발생할지 직관적인 다이어그램으로 알아봅시다.

<img width="903" height="434" alt="image" src="https://github.com/user-attachments/assets/1e9124e8-14aa-4d77-ad56-579deed15e03" />


*   **현업 실무자의 시선**: 실무에서 OOM(Out of Memory) 에러가 발생하면 무조건 Batch Size부터 절반으로 깎는 것이 국룰입니다. 반대로 GPU 자원이 남아돌면 Batch Size를 늘려 학습 속도를 극대화합니다. 통상적으로 2의 거듭제곱(32, 64, 128, 256)을 사용하는 이유는 메모리 버스트 아키텍처가 2의 배수 단위를 가장 빠르게 읽어오기 때문입니다.

### 2. 왜 float64가 아니라 float32인가?
파이썬의 기본 실수형은 64비트(`float64`)입니다. 하지만 우리는 전처리 단계에서 강제로 32비트(`float32`)로 다운캐스팅을 했습니다.
*   **현업 실무자의 시선**: 딥러닝은 엄청난 정밀도가 필요한 우주선 궤도 계산이 아닙니다. 대략적인 방향성(확률)만 맞으면 됩니다. 최신 엔비디아(NVIDIA) 딥러닝 전용 GPU들은 32비트 혹은 16비트 연산(FP32, FP16)에 하드웨어적으로 미친 듯이 최적화되어 있습니다. 32비트를 쓰는 순간 데이터를 딱 절반의 용량으로 압축하므로, GPU RAM에 데이터를 두 배 더 많이 올려서 2배 더 빠르게 학습시킬 수 있습니다. 이것이 딥러닝에서 데이터 타입을 깎아내는 강력한 실무적(Business) 이유입니다.
